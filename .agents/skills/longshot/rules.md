# Longshot Rules

Cross-cutting rules, guards, and invariants that apply across longshot phases. When these conflict with phase-specific instructions, **these rules govern**. When in conflict with user instructions (CLAUDE.md / AGENTS.md / GEMINI.md or direct requests), **the user wins**.

Complements:
- [principles.md](principles.md) — the *why* (quality bar, engineering philosophy)
- [profiles.md](profiles.md) — the *where* (per-project commands, layers, agents)
- [SKILL.md](SKILL.md) — the *when* (orchestration, flags, pipeline overview)

---

## 1. Invariants

Rules that apply at every phase. Violation is a bug, not a judgment call.

### 1.1 Top-Level Build Commands Only

**MUST ALWAYS** use top-level `make` or `npm` commands — never invoke underlying tools directly (e.g., `golangci-lint`, `eslint`, `tsc`, `prettier`, `go vet`). Top-level commands include project-specific configuration, flags, and multi-step pipelines that direct tool invocations miss.

Examples:
- ✓ `make check-style`, `npm run check`, `npm run fix`
- ✗ `golangci-lint run`, `eslint .`, `tsc --noEmit`

Applies to Phases 3, 4, 5, 6, 7 — anywhere a quality tool runs.

### 1.2 Timestamp Format

All timestamps in `state.json` (and any other structured artifact) MUST include the full date AND time using:
- Bash: `date -u +%Y-%m-%dT%H:%M:%SZ`
- JS: `new Date().toISOString()`
- Go: `time.Now().UTC().Format(time.RFC3339)`

Never use date-only formats. `2026-03-19T00:00:00Z` is WRONG; `2026-03-19T14:32:45Z` is correct. Every `"timestamp"`, `"created"`, and `"updated"` field must capture the actual wall-clock time of the event.

### 1.3 No AI Attribution in Commits or PRs

- No `Co-Authored-By:` trailers naming an AI assistant, model, or CLI
- No "Generated with …" / "Generated by AI" / "Written by …" style lines referencing an AI tool
- No AI/assistant/model branding in commit subjects, bodies, or PR descriptions, regardless of which platform generated the work

The commit should read as if the engineer wrote it. A `Context: /longshot — <recap>` footer (using `state.json.recap`) is the only longshot-specific trailer permitted.

### 1.4 Artifacts Are Source of Truth

Plans, specs, and state are read from disk, never reconstructed from conversation context:

- **Phase 3** reads `<artifact_dir>/plan.md` from disk before implementing — never from what was drafted in conversation in Phase 2
- **Phase 4** reads the `## Test Plan` section from `plan.md` as its test specification
- **Phase 8** reads `state.json` for fix version, security classification, PR URL
- **`--continue` / `--skip-to`** rebuild full context from disk — they do not rely on prior conversation

This prevents context drift across phases and makes resumption deterministic.

**Corollary — reviewer/subagent artifacts must land under `<artifact_dir>`**: any agent, skill, or slash command that produces findings/review/analysis output MUST write to `<artifact_dir>/findings/phase<N>/...`, never to the repo working tree. For tools with rigid output paths (e.g., `/comprehensive-review:full-review` hardcodes `<cwd>/.full-review/`), the invoking phase redirects at source — pre-seed the output dir and symlink the rigid path into it, or intercept the stream and Write directly. After the command returns, verify `git status` shows no review-related entries. Cleanup is a fallback, not the primary strategy — leaked artifacts bloat diffs and risk accidental commits.

### 1.5 state.json Update Ritual

At the end of every phase, update `state.json` atomically with:
1. `phases["<N>-<name>"].status = "complete"`
2. `phases["<N>-<name>"].timestamp = <RFC3339 UTC>`
3. `current_phase = <N>`
4. `updated = <RFC3339 UTC>`
5. Record any phase-specific fields (e.g., `commits[]` additions, `security` updates, `release.fix_version`)

Phase docs reference this ritual by number — they do not restate the five-step procedure.

### 1.6 Checkpoint Commits at Gates

After Phases 2, 3, 4, 6 (the gated phases), create a checkpoint commit and record:

```json
{
  "phase": "<N>-<name>",
  "sha": "<commit sha>",
  "timestamp": "<RFC3339 UTC>"
}
```

in `state.json.checkpoints[]`. These SHAs are what `--revert <phase>` consumes.

## 2. Git Safety

Apply regardless of profile.

- **NEVER rewrite public history.** Once a PR is opened, `git rebase`, `git reset --hard`, and amends to pushed commits are off-limits — these rewrite history reviewers are tracking. Reversion on public branches uses `git revert` (preserves history), never `git reset --hard`.
- **Before the PR is opened**, `git rebase` IS acceptable (and often encouraged) for local cleanup — squash WIP commits, reword messages, reorder, drop noise. The distinguishing rule is *public* history, not rebase itself.
- **NEVER** force push (`-f`, `--force`, `--force-with-lease`) unless the user explicitly requests it.
- **NEVER** skip hooks (`--no-verify`, `--no-gpg-sign`) unless the user explicitly requests it. If a pre-commit hook fails, investigate and fix — don't bypass.
- **NEVER** `git add -A` or `git add .`. Stage specific files by name to avoid capturing `.env`, credentials, tracking files, or unrelated changes.
- **NEVER** unstage files without explicit user permission.
- **NEVER** amend a commit after a hook failure — the commit didn't happen; hook failure means `--amend` would modify the *previous* commit. Fix, re-stage, create a NEW commit.
- **ALWAYS** follow `PULL_REQUEST_TEMPLATE.md` if present in the target repo (check `.github/` and root). Fill every required section — do not skip, do not substitute a custom template. Only fall back to the default body ([phase-7-ship.md § Step 7.5](phase-7-ship.md)) when no template exists.
- **Commit format**: conventional (`<type>(<scope>): <subject>`), subject ≤50 chars, imperative mood, no period. Types: `feat|fix|docs|style|refactor|test|chore|perf`.

`--revert <phase>` uses `git revert` on the SHAs recorded in `state.json.checkpoints[]`. It preserves history.

## 3. Security Handling

Engages when Phase 1.1.5 sets `state.json.security.is_security_issue = true`. All rules below apply for the **remainder of the pipeline**, not just the phase that detected it.

Reference: [Mattermost Sensitive PRs Handbook](https://handbook.mattermost.com/operations/security/product-security/working-on-sensitive-prs).

### 3.1 Language Rules (applies to all artifacts)

The following terms **must not** appear in branch names, commit messages, code comments, test descriptions, PR titles, or PR bodies:

- Vulnerability class names: `XSS`, `CSRF`, `SSRF`, `SQLi`, `RCE`, `LFI`, `RFI`, `XXE`, `SSTI`
- Generic security terms: `vulnerability`, `exploit`, `attack`, `bypass`, `payload`, `injection`
- Identifiers: CVE IDs (`CVE-YYYY-NNNN`), CVSS scores, HackerOne report IDs
- The Jira ticket title verbatim (if it describes the vulnerability)

Reference the ticket by ID only: `See MM-1234` — never `Fix XSS in MM-1234`.

Use **functional language** instead:
- ✓ `fix(api): improve input handling in channel member endpoint`
- ✗ `fix: prevent XSS in channel member endpoint`

### 3.2 Branch Naming

Security branches use the ticket ID alone (e.g., `MM-1234`) or a neutral functional name (`MM-1234-input-handling`). If the branch was already created with a descriptive security term, rename it before the first push.

### 3.3 Test Code

Tests MUST use benign, functionally-equivalent inputs — never live PoC exploit payloads. A test named `TestRejectsMalformedInput` with input `{"user": "<invalid>"}` is correct; a test named `TestXSSProtection` with input `<script>alert(1)</script>` is wrong on both counts.

### 3.4 PR Sanitization

Before creating the PR:
- Title: functional description, no vulnerability class, no CVE, no ticket title verbatim
- Body: describe the fix functionally, omit vulnerability specifics
- Do NOT link HackerOne reports, PoC exploits, or security advisories in the PR body
- Keep discussion in Jira, not GitHub comments

### 3.5 Commit Log Audit (Phase 6)

Before push, run `git log <base_branch>..HEAD --oneline` and flag any subject or body matching the forbidden terms from §3.1. If found, advise the user to squash-rewrite locally before push (acceptable pre-push; very painful post-push).

### 3.6 Leak Response

If sensitive language reaches a public PR accidentally:
1. Edit the PR title/body immediately via `gh pr edit`
2. Force-push a corrected history **only** if the user authorizes it (see §2 — this is an explicit-authorization exception)
3. Notify the Security team

### 3.7 MUST_FIX Discipline

All findings from Phase 6.7 (security review) on a security issue are MUST_FIX. Phase 7 does not proceed with open MUST_FIX items from a security review, regardless of the usual 2-round budget.

### 3.8 Backport Timing

Security backport PRs may be embargoed. Do NOT open backport PRs in Phase 8 until the Security team gives the go-ahead. Coordinate release timing across branches.

## 4. Retry & Escalation Budgets

Each gate has a bounded retry budget. When exceeded, STOP per §6 — do not silently retry.

| Phase | Operation | Budget | Escalation |
|-------|-----------|--------|------------|
| 2 | Plan review iterations | 2 | STOP if 3+ MUST_FIX remain |
| 3 | Auto-review rounds | 2 | Report remaining MUST_FIX, proceed with warning |
| 4 | Test fix attempts per error signature | 3 | STOP — report `{test_file, error_type, signature, attempts, last_error, suggested_fix}` |
| 5 | Lint/type fix attempts | 2 | Report and proceed with warning |
| 6 | Code review rounds | 2 | Report remaining MUST_FIX; proceed with warning (but see §3.7 for security) |
| 7 | Pre-ship verification attempts | 2 | Proceed with warning noted in PR body |
| any | Convergence rounds (swarm) | 3 | Escalate on: not decreasing, oscillation, or count unchanged |

**Per error signature, not per test file.** If `TestFoo` and `TestBar` both fail with the same hash of `error_type + error_message`, that counts as one signature, not two. This prevents retries from masking a systemic bug.

**Transient failures**: log the signature, retry once. If the same signature recurs across 2 separate runs, reclassify as `CODE_BUG` — retries must not silently mask race conditions.

## 5. Agent & Tool Fallback

Longshot degrades gracefully. Missing dependencies trigger a warning, not a halt (except `git`, which is required).

### 5.1 Agent Fallback

All domain agents are **soft references**. If an agent isn't registered in the platform's agent directory (e.g., `~/.claude/agents/`, project-level `.claude/agents/`, or the equivalent on other platforms):
- The phase proceeds without that review dimension
- A note is added to `<artifact_dir>/findings/synthesis.md`: `"<agent-name>: unavailable, skipped"`
- Do NOT attempt to emulate the agent inline — skip the dimension

### 5.2 Team Spawn Fallback

If `TeamCreate` is unavailable or a team spawn fails:
- Drop to leader-only execution for that phase
- Log the fallback in `state.json.phases["<N>"].notes`
- Record `execution_mode: "solo-fallback"` for that phase

### 5.3 CLI Tool Fallback

From Phase 0 toolchain probe:

| Tool | Missing behavior |
|------|-----------------|
| `git` | **ABORT** — pipeline cannot run |
| `gh` | Phase 7 PR creation disabled — suggest `--skip pr`; Phase 8 linking degrades |
| `acli` | Phase 1/8 Jira/Confluence steps fall back to Atlassian MCP, then to manual prompts if both are missing |
| `make` | Warn (MM profile) — document that quality commands will fail |
| `npm`/`node`/`go` | Warn — profile-specific commands fail, note in report |

### 5.4 MCP Fallback

| MCP | Missing behavior |
|-----|------------------|
| Playwright MCP | Phase 4.6 falls back to printing a manual checklist |
| Figma MCP | Phase 1 skips design-context fetching |
| Atlassian MCP | `acli` is the primary tool for Jira/Confluence; MCP is only used as fallback when `acli` is unavailable. If both are missing, Phase 1/8 degrade to manual prompts. |

## 6. STOP Protocol

Used at every pipeline halt: gate failure after max retries, context limit approach, irrecoverable error.

**Standard STOP message format:**

```text
LONGSHOT STOPPED at Phase <N> (<phase-name>)
Reason: <what went wrong>
Last output:
  <last 5 lines of failing command output, or failing test file + line number>
State: <artifact_dir>/state.json
Resume: /longshot --continue (or --skip-to <next-phase>)
Revert: /longshot --revert <current-phase>
```

Update `state.json`:
- `status = "stopped"` (not `"complete"`)
- `phases["<N>-<name>"].status = "failed"` (or `"partial"`)
- `phases["<N>-<name>"].stop_reason = "<reason>"`
- `updated = <RFC3339 UTC>`

**Context-limit STOP** is special: checkpoint state first, then emit the message. Suggest the user run `/longshot --continue` in a **new session** (not the current one — the current session is about to lose context).

## 7. Swarm Mode: File Ownership & Convergence

Applies whenever a phase spawns a team (Phases 2, 3, 4, 6 when `TeamCreate` is available).

### 7.1 File Ownership

- **One writer per file at a time.** The leader assigns ownership per layer (model, store, app, api, webapp) and enforces the boundary.
- If two agents need the same file: either serialize the work or the leader merges.
- **Only the leader touches git** — stash, commit, branch, and SHA recording happen in the leader context.

### 7.2 Shared Findings Directory

Agents write to `<artifact_dir>/findings/`, namespaced by pipeline phase:

```text
findings/
└── phase<N>/              # e.g., phase2/, phase3/, phase6/
    ├── <source>/          # Optional: raw per-agent/per-dimension output
    ├── round-1/           # Archived convergence rounds
    ├── round-2/
    └── synthesis.md       # Final merged report for this phase (~30 lines)
```

**The leader reads ONLY the phase's `synthesis.md`.** Full findings are preserved for audit but never pulled into the leader's context window.

### 7.3 Interface Summary Broadcasts

After each implementation layer completes, the leader broadcasts to all remaining agents:

```text
INTERFACE SUMMARY from <completed-layer>:
- Exported types/structs: <list with signatures>
- Public functions: <list with signatures>
- API endpoints: <routes, methods, types>
- Database schema changes: <tables, columns>
```

This keeps downstream agents in sync without giving them full source access.

### 7.4 Convergence Loop

For review phases (3 auto-review, 6 standalone):
- Track MUST_FIX count across rounds
- Archive round findings to `findings/round-<N>/`
- **Exit conditions**: 0 issues (success) · count not decreasing (escalate) · oscillation detected (escalate) · max 3 rounds

## 8. Principle Applications

Connective tissue between [principles.md](principles.md) and phase behavior. A principle is cited inside a phase because it's **load-bearing** for a decision at that moment — not as decoration.

| Phase | Principle | How it applies |
|-------|-----------|----------------|
| 2 (Plan) Step 2.4 | *"AI makes iteration cheap — use it to validate designs or approaches."* | Devil's advocate review is cheap to run; use it to surface simpler alternatives before committing. If two approaches seem viable, prototype the riskier parts of each before choosing. |
| 3 (Implement) Step 3.3 | *"Accountability includes correctness, stability, security, and operational readiness, not just passing tests."* | Every implementation decision must be defensible — not just working, but production-ready for real customer deployment. Green tests are necessary, not sufficient. |
| 4 (Test) Step 4.6 | *"Accountability includes correctness, stability, security, and operational readiness, not just passing tests."* | Scripted tests verify code correctness; exploratory testing verifies feature correctness. Both required. |
| 6 (Review) | *"Optimize for PRs that are easy to review, through small scope, clear intent, and strong self-review."* | Phase 6 is the strong self-review — every finding should be addressed before a peer sees the code. Reviewers are not a safety net for AI-generated sloppiness. |
| 7 (Ship) | *"Predictable, smooth processes beat raw speed."* | Fewer well-crafted PRs beat many sloppy ones. This PR should be appropriately sized, strongly self-reviewed, and a point of pride — not a burden for peers. |

Phases reference this table rather than restating the principle inline. When the principle is the reason for a step, cite `rules.md §8` and move on.

## Rule Precedence

When rules appear to conflict:

1. **User instructions** (CLAUDE.md, AGENTS.md, direct user requests) — highest
2. **rules.md** (this file) — overrides default phase behavior
3. **Phase files** (phase-N-*.md) — the baseline
4. **principles.md / profiles.md** — context and justification

If a phase file contradicts this file, this file wins. If the user contradicts this file, the user wins.
